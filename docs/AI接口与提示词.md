# AI 接口与提示词

## 接口约定

当前项目对接三类模型服务，均按“OpenAI 兼容接口”组织主要参数：

- DeepSeek
- Kimi
- 千问

各服务的主要配置项包括：

- `base_url`
- `api_key`
- `model`
- `timeout_s`

其中：

- 配置文件默认写入用户目录（Windows 为 `%APPDATA%/sj_generator`），不写入仓库目录
- 配置文件当前保存 `base_url / model / timeout_s` 等非密钥字段
- `api_key` 当前从环境变量读取，并通过程序设置界面写入用户环境变量
- DeepSeek 额外区分导题模型与解析模型，解析模型默认值为 `deepseek-reasoner`
- 各厂商配置页支持维护“可用模型列表”；项目配置中的模型选择来自这些列表
- 千问预设模型列表包含 `qwen3.6-plus`

建议调用 chat completion 类接口，要求模型输出严格 JSON，便于程序解析入库。

## 资料解析任务（txt/docx -> 结构化题目）

目标：从资料文本中抽取选择题，输出为 JSON 数组，每个元素代表一道题。当前导题流程的题号题型解析模型组、题目内容解析模型组、轮次与比例均来自 API 配置中的项目配置。

### JSON 输出结构（当前结构）

每题字段：

- id：题目编号（字符串或数字，可为空）
- stem：题目/题干（字符串）
- option_1：第 1 个选项正文
- option_2：第 2 个选项正文
- option_3：第 3 个选项正文
- option_4：第 4 个选项正文
- choice_1 ~ choice_4：可转多选时 A ~ D 对应的数字映射；非可转多选为空字符串
- answer：答案（字符串，单选如 A，多选如 ACD）

### 提示词骨架（建议）

系统/指令：

- 你是一个题库整理助手
- 你必须输出严格 JSON，不要输出多余文本
- 识别单选与多选
- 若原文选项使用 ①②③…，请按出现顺序映射为 A,B,C...
- 答案必须使用 A-Z 字母，单选为单字母，多选为按字母升序拼接

用户内容：

- 原始文本（从 txt/docx 提取的纯文本）

### 一致性策略（当前实现）

- 题号题型解析与题目内容解析都按项目配置中的模型组执行
- 最高轮次由项目配置决定，当前支持 `1 ~ 8` 轮
- 通过比例由项目配置决定
- 未返回也按不一致处理，不会单独降低分母标准

### 容错策略（建议）

- 若题干与选项边界不清晰，尽量按“最可能结构”抽取
- 若无法确定答案，answer 置空字符串，并在 analysis 标注“原文未提供明确答案”
- 保留原文关键信息，不要自创不存在的选项内容

## 解析生成任务（补解析）

输入：`stem`、`options`、`answer`

输出：`analysis`（中文，简洁说明选项为什么对/错）

要求：

- 当前解析生成支持选择 `DeepSeek / Kimi / 千问`
- 可结合 `reference/resource/` 下的全部 md 作为参考资料
- 可结合常见错题归因参考文件辅助生成
- 解析生成并发数由 API 配置中的“答案解析生成并发数”决定
